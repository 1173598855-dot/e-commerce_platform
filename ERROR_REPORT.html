#1 网关路由剥离逻辑错误 — 大部分 API 路由不可达 Critical

网关使用 app.use(prefix, handler) 注册路由，Express 会自动剥离挂载前缀。但下游微服务的路由定义包含了该前缀子路径，导致剥离后的 URL 无法匹配任何路由。

问题文件
backend/microservices/gateway/index.js (L170-186, L188-210) 
网关: app.use("/api/categories", handler)
请求: GET /api/categories → req.url 被剥离为 "/"
转发: product-service:4002/ → 命中 app.get("/") → 返回商品列表（非分类！）

同理以下路由全部失效：
  /api/categories  → product-service 期望 /categories，收到 /
  /api/reviews/*   → product-service 期望 /reviews/*，收到 /*
  /api/skus/*      → product-service 期望 /skus/*，收到 /*
  /api/addresses/* → user-service 期望 /addresses/*，收到 /*
  /api/favorites/* → user-service 期望 /favorites/*，收到 /*
  /api/coupons/*   → user-service 期望 /coupons/*，收到 /*
  /api/points/*    → user-service 期望 /points/*，收到 /*
  /api/chat/*      → mq-service 期望 /chat/*，收到 /*
  /api/merchants/* → user-service 期望 /merchants/*，收到 /*
  /api/payments/*  → order-service 期望 /payment/*，收到 /*
  /api/logistics/* → order-service 期望 /logistics/*，收到 /*
修复方案
方案A（推荐）：网关不剥离完整前缀，只剥离 /api 或 /api/v1，下游服务路由保持 /categories、/reviews 等。
方案B：下游服务去掉路由中的子路径前缀（如 /categories → /），与网关剥离后的路径对齐。
方案C：网关使用 http-proxy-middleware 替代手动代理，自动处理路径重写。

#4 product-service 路由顺序错误 — /:id 拦截 /categories Critical

Express 按路由定义顺序匹配。app.get("/:id") 定义在 app.get("/categories") 之前，导致 /categories 被 /:id 拦截（id="categories"），返回 "商品不存在" 而非分类列表。

问题文件
backend/microservices/product-service/index.js 
L50: app.get("/:id", ...)       ← 先定义，拦截所有单段路径
L70: app.get("/categories", ...) ← 后定义，永远无法匹配
L77: app.get("/categories/all", ...)
L84: app.get("/categories/:id", ...)

被拦截的路径:
  /categories    → 命中 /:id (id="categories") → 查询商品失败 ❌
  /hot           → 命中 /hot (定义在 /:id 之前) → 正常 ✓
修复方案
将所有静态路径路由（/categories, /categories/all, /reviews/*, /skus/*）移到 /:id 之前定义。
#5 前端与后端响应格式不匹配 — Token 刷新逻辑永远不触发 Critical

前端期望 code 为数字类型（10000-50000），但后端返回的 code 是字符串（"OK"、"TOKEN_INVALID"）或根本不包含 code 字段。导致前端的成功/错误判断逻辑和 Token 自动刷新机制完全失效。

前端代码
frontend/src/api/request.js (L129, L159) 
L129: if (data && typeof data.code === 'number') {  // 期望数字
L159: if (serverCode === ERROR_CODES.TOKEN_EXPIRED)   // 30001 数字

前端 ERROR_CODES: { TOKEN_EXPIRED: 30001, ... }
后端实际返回
gateway/index.js (L30-46) / 各服务 shared/index.js (L5-11) 
网关 respondSuccess: { code: "OK" }              // 字符串
网关 respondError:   { code: "TOKEN_INVALID" }      // 字符串
服务 sendRes:        { success, data, message }     // 无 code 字段
服务 sendError:      { success: false, message }    // 无 code 字段
修复方案
统一响应格式：后端全部使用数字错误码（与 README 中定义的 10000-50000 码段一致），或者前端改为匹配字符串错误码。建议统一为数字码并补全 code 字段。
#6 auth-service 缺少 /refresh 端点 — Token 刷新永远失败 Critical

前端调用 POST /auth/refresh 刷新 Token，但 auth-service 没有定义 /refresh 路由。Token 过期后用户会被直接登出，无法自动续期。

前端调用
frontend/src/api/request.js (L112) / frontend/src/api/index.js (L13) 
refreshApi.post('/auth/refresh', { refreshToken })
后端缺失
backend/microservices/auth-service/index.js 
已定义的路由: /register, /password-login, /send-code, /sms-login,
              /wx-login, /qq-login, /profile (GET/PUT)
缺失的路由: /refresh ❌
修复方案
在 auth-service 中添加 POST /refresh 端点，验证 refreshToken 并返回新的 accessToken。

#7 SMS 验证码后门 — Redis 不可用时任意验证码 "1234" 可登录 High

当 Redis 连接失败时，auth-service 的验证码登录回退为硬编码验证码 "1234"。在生产环境中，如果 Redis 意外宕机，攻击者可以用任意手机号 + "1234" 直接登录或注册新账号。

问题代码
backend/microservices/auth-service/index.js (L96-98) 
if (redisClient) {
  const storedCode = await redisClient.get(`sms:${phone}`);
  isValidCode = storedCode === code;
} else {
  isValidCode = code === "1234";  // ← 后门！
}
修复方案
Redis 不可用时应直接拒绝验证码登录（返回 "验证码服务不可用"），而非使用硬编码后门。
#8 seed.sql 数据与 Schema 不匹配 — 导入会失败 High

种子数据脚本引用了 Schema 中不存在的列，导入时会报错。

问题1: hot_searches 表插入了不存在的列
database/seed.sql (L99) vs database/schema.sql (L319-326) 
seed.sql:  INSERT INTO hot_searches (keyword, search_count, is_hot, sort_order) ...
schema:    无 is_hot 列, 无 sort_order 列
问题2: 测试用户密码 hash 被截断
database/seed.sql (L111-113) 
seed.sql:  '$2a$10$N9qo8uLOickgx2ZMRZoMye'  (30字符)
正确长度:  '$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy' (60字符)
影响:      测试用户无法用密码 "123456" 登录，bcrypt.compare 必定失败
修复方案
1. 删除 seed.sql 中 hot_searches 的 is_hot 和 sort_order 列
2. 用完整的 bcrypt hash 替换截断的密码（可通过 bcrypt.hash('123456', 10) 生成）
#9 search-service hot_searches ON DUPLICATE KEY 无唯一约束 High

search-service 使用 ON DUPLICATE KEY UPDATE 来累加搜索次数，但 hot_searches 表的 keyword 列没有 UNIQUE 约束，该语句永远不会触发更新分支，每次搜索都会插入新行。

问题代码
backend/microservices/search-service/index.js (L61) vs database/schema.sql (L319-326) 
代码:  INSERT INTO hot_searches (keyword, ...) VALUES (?, ...)
       ON DUPLICATE KEY UPDATE search_count = search_count + 1
Schema: keyword 列无 UNIQUE 约束 → ON DUPLICATE KEY 永远不触发
修复方案
在 schema.sql 中为 hot_searches.keyword 添加 UNIQUE KEY uk_keyword (keyword)。同时修复 search-service INSERT 语句中引用的不存在的 is_hot 列。
#10 Nginx SSL 证书目录为空 — Nginx 无法启动 High

Nginx 配置要求 SSL 证书文件 /etc/nginx/ssl/server.crt 和 /etc/nginx/ssl/server.key，docker-compose 挂载了 ./nginx/ssl 目录，但该目录为空。Nginx 容器启动时会因找不到证书而崩溃。

问题文件
nginx/nginx.conf (L50-51) / docker-compose.yml (L212) 
nginx.conf:  ssl_certificate /etc/nginx/ssl/server.crt;
             ssl_certificate_key /etc/nginx/ssl/server.key;
实际状态:    nginx/ssl/ 目录存在但为空
修复方案
生成自签名证书放入 nginx/ssl/（开发环境），或在文档中说明需要手动放置正式证书。可添加 openssl req -x509 -newkey rsa:2048 -keyout nginx/ssl/server.key -out nginx/ssl/server.crt -days 365 -nodes -subj "/CN=localhost" 到部署脚本。

#15 网关代理对 GET 请求发送了 JSON body Medium

proxyRequest 函数中 req.body 在无 JSON body 时可能是 {}（空对象），JSON.stringify({}) = "{}"（长度2），会对 GET 请求也发送 body。

问题文件
backend/microservices/gateway/index.js (L116) 
const body = req.body ? JSON.stringify(req.body) : "";
// 当 req.body = {} 时，body = "{}"，body.length = 2 > 0
// → GET 请求也会被写入 body
修复方案
检查 req.method 和 Object.keys(req.body).length，仅在 POST/PUT/PATCH 且有实际 body 时才序列化。
